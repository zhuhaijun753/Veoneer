This document aims to describe how to design the vehicle bus interface for the reference products.

1 DBC format

Note that for DBC files Vector have limited the name length to 32 characters for messages, signals and enums, if the wanted name is too long try to shorten it in a way that is easy for a human to read.

2 Version handling

The vehicle specification shall contain a message called InterfaceVersion. This message shall contain MajorVersion and MinorVersion as U8. MajorVersion shall be incremented on breaking changes and MinorVersion for all other changes. Recommended cycle time is 1000 ms so that it is hard to make a recording without it. The current version is defined using the default value in the DBC.

3 Messages

Transmitter shall be set to the transmitting ECU.

3.1 Attributes

  • Send type: cyclic, spontaneous, etc
  • Cycle time: for cyclic messages, defined by the internal data rate

3.3 Content

As a lot of small messages creates overhead it is preferred that output that relates to the same feature/function be grouped together. Signal groups can be used to separate different signals from each other or clarify that some signals are associated within a message.

Comment: Should describe the content of the message

3.5 Priority

Depending on the bus/vehicle interface (CAN, FlexRay, Ethernet) the priority of the output messages can be controlled. Messages can be divided in the following categories.

3.5.1 Time critical

Typically actuator requests or data for a time critical feature. E.g. brake request from AEB or object detection for AEB (crossing cyclists).

3.5.2 Important

Convenience features and warnings. E.g. Headlight control or Lane Deviation Warning.

3.5.3 Information

Data that is displayed to the driver. E.g. Traffic signs.

3.5.4 Diagnostics

UDS, measurement data

4 Signals

4.1 Naming convention

Names are written in camelCase.

4.2 Endianess

The current microcontrollers are little endian/Intel based, therefore this byte order shall be used for signal encoding. Deviations are only allowed when a standard that is used requires otherwise, the deviations shall be noted and motivated in the Comment field.

4.3 Encoding

To optimize the bus usage the following principles shall be applied.

4.3.1 Decimal values

Shall be sent as scaled integers using the Factor option.

4.3.2 Intervals

If an equally positive and negative interval is needed a signed value can and should be used. E.g. yaw rate.

If an unequal interval can be applied an unsigned integer with offset shall be used. E.g. speed.

4.3.3 Enumerations / Bitmasks

Both enumerations and bitmask signals shall be defined using value tables. This is so that tools can visualize the data.

4.4 Documentation

The DBC/arxml format supports at least the following fields that are expected to be filled in.

Field Description
Unit Self explanatory
Init value Default value that is applied at startup until a calculated value is provided
Comment A short description of the signal if the name is not self explaining

5.2 Multi message

Lists that need multiple messages to be transmitted shall be separated into header and content message, where the content message is repeated on the vehicle bus to produce the complete list. Common things for the list shall be contained in the header, e.g.Â timestamp.

Each object in the list shall be packaged exactly the same, be structured in a signal group and contained in the content message. The signal and group names shall have names ending with an alphabetized index based on the index within the message. E.g. ObjectA, ObjectB, â¦

The header shall contain the length of the object list and index synchronizing the Header and Content, the signal shall have a name ending with ListLength and ListIndex respectively. E.g. objectsListLength and objectsListIndex.

If using the same message ID for all objects each message shall contain the message index so that the list/array can be built up in the receiving end, this signal shall have a name ending with Index. E.g. objectsIndex.

If using different message IDs for objects then each message ID shall be associated with a message index. In this case the Header is assigned the lowest ID and then the following messages are added in list order. E.g. Header 0x220, Content0 0x221, Content1 0x222, â¦

To synchronize the header and content messages a signal ending with ListIndex shall be present in both header and content message. E.g. Header: objectsHeaderListIndex, Content: objectsListIndex.
